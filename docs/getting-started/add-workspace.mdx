## Overview

A workspace is the top-level project ctx records Work for. It can be one repo,
a directory that contains multiple repos, or a scratch workspace created under
the ctx data root.

This is the main unit for settings, tasks, sessions, Work records, and review.
The desktop Workbench and `ctx work` CLI read from the same local workspace
registry under `CTX_DATA_ROOT` or `~/.ctx`.

## Local workspace flow

1. Open the workspace list in the app.
2. Add your repository root or the parent directory for a multi-repo workspace.
3. Open the workspace.
4. Verify settings before starting a real task.

For CLI-only setup checks, register or select the workspace in the same data
root and then use:

```bash
ctx setup status
ctx work schema
ctx work search "validation"
```

If the data root contains more than one workspace, pass `--workspace-id` to
Work commands or run them from inside the registered workspace root. The older
`ctx work list` and `ctx work export` commands inspect/export compatibility
change-set records; use [Work Records](/work-records) for the current Work
Record observability commands.

## Scratch workspace

A scratch workspace is an empty git repo under the user-local ctx data root. It
is useful for trying the Workbench without pointing ctx at an existing checkout.
It is not sandboxed by default. Container isolation is an ADE execution mode you
choose separately for containerized runs.

## Remote workspace flow

Use this when the repository lives on your own VPS, devbox, or remote machine.

1. Start the workspace wizard and choose a remote workspace.
2. Enter the remote host in `user@host` form.
3. Verify the SSH connection.
4. If prompted, enter a one-time SSH password so ctx can install key-based SSH auth. The password is not stored.
5. Select the remote repository root and finish setup.

Before you start, make sure the repo already exists on that machine and your SSH account has access to it.

## Configure baseline before real work

- Provider/harness authentication
- Model defaults
- Execution mode for ADE runs (host/container)
- Network egress policy for containerized ADE runs
- Worktree bootstrap setup if each task needs repeatable setup
- Merge queue settings only if you plan to run parallel tasks and land changes through the queue

## Workspace sanity checks

- Workspace path is correct.
- `ctx setup status` sees the expected workspace when using the same data root.
- You can create a task in this workspace.
- Session runs can start.
- Diff review works after a test edit.
- For remote workspaces, reconnecting to the same host reopens the expected workspace cleanly.
