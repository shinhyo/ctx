## Overview

A task should represent one human objective.

Tasks are your main planning and review boundary in ctx. A task is the human
objective and the unit of delegated agent work; sessions, runs, worktrees,
change sets, artifacts, checks, and pull requests hang off that objective
instead of replacing it. Gates belong to the change set, pull request, ref, or
merge candidate being evaluated, not to a local agent session.

## How tasks relate to Work

A task may need one or more agent sessions. Each session may contain several
runs, such as an initial pass, a revision pass, and a validation pass. Those runs
work in isolated worktrees and produce reviewable change sets.

Use the task to answer "what outcome are we trying to achieve?" Use the session
and run history to answer "how did the agent attempt it?" Use the change set,
artifacts, and checks to answer "what evidence do we have?" Use gates to answer
"can this change set or pull request advance?"

## Tasks and pull requests

Tasks and pull requests are many-to-many.

One task can produce multiple pull requests when review should be split by risk,
ownership, or release timing. One pull request can advance multiple tasks when
the final changes are best reviewed together.

The connection should be explicit: the relevant sessions, runs, change sets,
artifacts, and checks contribute to the task or pull request. Avoid relying on a
hidden one-task-one-PR assumption when planning review.

## Writing strong task titles

Good titles are specific and reviewable:

- `Fix OAuth callback redirect loop`
- `Add empty-state for archived projects`
- `Refactor provider selector rendering`

## First message checklist

Include in your first prompt:

- expected behavior
- constraints
- validation requirements

This keeps sessions aligned and reduces scope drift.

## Simple task template

```text
Goal: <one concrete outcome>
Constraints: <what must not change>
Validation: <tests or checks to run>
Output: summarize changed files and rationale
```

## When to split a task

Split when:

- the change spans unrelated areas
- review becomes too large to reason about
- different risk levels need separate review decisions

## Lifecycle hygiene

- keep active task list focused
- archive completed tasks
- avoid reviving old tasks for unrelated work
- split or link tasks and pull requests when review boundaries change
