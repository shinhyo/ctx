# Work and ADE Extension Decisions

This page records the public architecture decisions behind the local Workbench
composition work in this branch.

The goal is to make ctx a programmable, local-first ADE for coding agents:
tasks, sessions, runs, change sets, artifacts, checks, and plugin contributions
should be visible through reusable UI primitives instead of being locked into
one hard-coded workbench layout.

## Public Scope

This public repo should contain:

- local Work/ADE records and projections;
- local schema validation and import/export shapes;
- Workbench UI primitives and built-in templates;
- local plugin manifest, inventory, command, provider-adapter, and future UI
  contribution foundations;
- redaction and provenance fields that make sharing explicit and user-controlled.

The `ctx work` CLI should be public and documented. It is an advanced surface,
but it is important because it proves the ADE and standalone Work capture use
the same local records.

This public repo should not contain hosted/team control-plane product surfaces,
billing, tenant administration, hosted audit dashboards, or organization policy
product flows. Those can reuse public records from a separate product layer, but
they should not shape this local-first implementation.

## Naming

Use **Work** as the product noun in UI and CLI copy.

The current v1 schema/API substrate in this branch still uses `agent-work`
names and paths. Treat those as the current compatibility surface for this
branch, not as permission to introduce a second competing vocabulary. A future
rename to `work` should be planned as one compatibility migration across CLI,
schemas, routes, TypeScript types, Rust types, and docs.

The public product noun is settled as **Work**. The `agent-work` namespace should
only remain as a compatibility surface until the migration can be done as one
planned change.

## Workbench Primitives

A Workbench primitive is a typed, reusable surface backed by ctx-owned state and
actions. Primitives may move between templates without changing task/session
semantics.

Initial primitives are:

- task list and task cards;
- active task/session;
- Work summary and change-set detail;
- artifacts/review evidence;
- terminal;
- empty/new-task composer state;
- provider/status surfaces.

Near-term product work should pursue ADE extensibility and Work capture in
parallel. The public wedge is the hackable ADE; Work capture is the substrate
that makes those extensions durable, replayable, exportable, and useful outside
the GUI.

Primitives must:

- read from stable Workbench/Work state;
- expose typed props/actions;
- preserve existing task/session behavior when moved;
- degrade to empty or read-only states when context is absent;
- avoid direct mutation of core Work records outside approved actions.

## Built-In Templates

Built-in templates prove the primitive model:

- **Classic**: the current default workbench behavior.
- **Kanban**: task lanes plus selected active task/session detail.
- **Multipane**: resizable split panes that can show active task, empty/new-task,
  Work detail, and terminal primitives.
- **Review**: focused task review with Work metrics, PR links, change sets, and
  active task content.

Template state is a local UI preference per workspace/window:

```text
WorkbenchTemplateState
  id: classic | kanban | multipane | review | plugin:<id>/<template>
  version: number
  layout: template-owned JSON decoded by ctx
```

Template state must not change Work records, provider state, or session history.

Plugin-compatible template IDs are host-owned. The initial supported vocabulary
is `classic`, `kanban`, `multipane`, `review`, and `review-summary`.
Unsupported plugin template IDs should render as source-labeled unsupported
contributions and fall back to a built-in template without mutating active
sessions.

## Plugin Direction

Use **plugin** for an installable bundle. A plugin may contribute one or more
extensions.

The current branch provides the local plugin substrate: manifest validation,
inventory, contribution registry, process command execution, provider adapter
registration, explicit reload, and auto-reload checks.

That is not yet a full user-owned ADE extension system. The next API layer
should add:

- typed TypeScript SDK;
- plugin dev/watch mode;
- plugin host lifecycle and diagnostics;
- permission/capability semantics;
- Workbench panel/template/renderer contribution APIs;
- source-labeled command palette actions.

Provider slash commands are provider-owned. Local ctx/plugin commands may be
bridged while the command palette work is still maturing, but they must remain
source-labeled and must not shadow provider-owned commands once provider command
catalogs are available.

Use ACP as the public provider integration protocol. A ctx plugin may
package/register an ACP-compatible provider, but ctx should not ask agent authors
to implement a competing public ctx agent protocol. CRP may remain an internal or
native compatibility layer where it is useful for existing providers and richer
runtime integration, but it should not be the public extension target.

Design future extension surfaces so they can include optional coding-agent
harness primitives without making ctx define a competing agent protocol. The
preferred public shape is: custom harnesses speak ACP, plugins package/register
them with ctx, and ctx Work events capture their transcripts, tool calls,
artifacts, checks, and contributions.

The harness starter kit should take inspiration from modular, reloadable,
user-owned coding-agent designs while staying deliberately unopinionated:
developers should be able to use ctx primitives for
shell/file/edit/artifact/transcript/work capture individually, or compose them
into a full coding-agent harness.

## Harness Starter Strategy

The harness starter is a future implementation convenience, not a protocol. Its
public shape is an ACP-speaking coding-agent harness that can be packaged as a
ctx plugin provider contribution.

The starter should help developers build a modular local coding agent while
keeping ownership of the agent loop, prompts, model routing, tools, memory, and
release process. ctx may provide optional local primitives, but a harness must
remain valid when it uses only ACP plus its own implementation choices.

Design goals:

- expose a narrow ACP adapter boundary for initialization, prompt turns,
  streaming content, tool calls, slash commands, sessions, auth, terminals, and
  cancellation;
- keep agent-loop policy outside ctx-owned code;
- make local primitives individually adoptable, so a harness can use ctx shell
  without ctx edit capture, or ctx transcript capture without ctx sandbox;
- preserve Work as the durable capture substrate, not as a required in-process
  agent API;
- package local development through plugins, so discovery, diagnostics, reload,
  and provider ownership follow the same rules as other ACP providers.

Non-goals:

- no competing `ctx agent` protocol;
- no hosted/team/enterprise harness surface in this public slice;
- no trademark or product positioning around other coding-agent projects;
- no assumption that ctx owns provider slash commands or agent internals;
- no direct Work store writes from harness code.

Potential local primitive boundaries:

| Boundary | Future package/module shape | Responsibility |
| --- | --- | --- |
| ACP adapter | `@ctx/harness-acp` | JSON-RPC stdio transport, ACP request/response typing, stream framing, diagnostics |
| Agent loop interfaces | `@ctx/harness-core` | turn lifecycle interfaces, cancellation tokens, event normalization, policy-free composition |
| Shell | `@ctx/harness-shell` | optional execution requests through ctx-approved shell/sandbox surfaces |
| Files | `@ctx/harness-files` | optional workspace file read/list/write mediation and provenance |
| Edits | `@ctx/harness-edits` | optional patch application and diff/change-set capture helpers |
| Sandbox | `@ctx/harness-sandbox` | optional sandbox descriptors, materialization hooks, and permission prompts |
| Transcript | `@ctx/harness-transcript` | optional message/chunk/tool-call event builders for Work capture |
| Artifacts | `@ctx/harness-artifacts` | optional artifact attachment metadata and redaction defaults |
| Work capture | `@ctx/harness-work-capture` | optional import/capture client that submits approved Work event payloads |
| Plugin packaging | `@ctx/harness-plugin` | manifest helpers for registering the ACP provider contribution |

These names are design hooks, not committed package names. Implementation may
start as repo-local modules or examples, but the boundaries should remain
separable so the starter does not distort the ADE, plugin, or provider runtime
architecture.

Example layout:

```text
examples/harness-starter/
  agent/
    loop.ts
    policy.ts
    tools.ts
  acp/
    server.ts
    sessions.ts
  ctx/
    plugin.ts
    work-capture.ts
    shell.ts
    artifacts.ts
  plugin.json
```

See [Harness Starter Boundaries](./examples/harness-starter-boundaries.md) for a
more concrete example.

## Local Data Rules

Private Work data should live under the user data root by default, not inside a
repository:

```text
~/.ctx/
  db/workspaces/<workspace-id>/db.sqlite
  objects/
  plugins/
  logs/
```

Repo-local files are opt-in only. Shareable exports should omit or redact
host-local roots and raw transcripts unless the user explicitly includes them.

## Remaining Public Work

The next public implementation steps are:

1. Follow the [Work Namespace Compatibility](./work-namespace-compatibility.mdx)
   contract for public `Work` naming and `agent-work` compatibility.
2. Follow the
   [Work Source Of Truth Contract](./work-source-of-truth-contract.mdx) so the
   ADE, `ctx work`, plugins, import/export, and future harness primitives use
   the same records.
3. Finish the `ctx work` CLI/export/import layer over the same records.
4. Add ACP provider plugin support according to the
   [ACP Provider Plugin Contract](./acp-provider-plugin-contract.mdx).
5. Build plugin contributions according to the
   [Plugin Contribution Contract](./plugin-contribution-contract.mdx).
6. Add provider command catalog/execution passthrough so provider commands stay
   provider-owned.
7. Move plugin commands into source-labeled command palette/workflow surfaces.
8. Add declarative Workbench panel, template, and renderer contributions.
9. Add plugin dev hot reload/watch mode.
10. Add import/export/redaction tests and examples.
11. Add a follow-on ACP-compatible coding-agent harness starter kit that uses
   ctx Work capture, sandbox, shell, file, artifact, and transcript primitives.
