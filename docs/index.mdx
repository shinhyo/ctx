ctx is a local Work record system for coding agents. It keeps the context around
agent-produced changes together: the objective, provider session, transcript,
commands, artifacts, diff, pull-request links, and validation evidence.

The desktop ctx Workbench is an optional Agentic Development Environment (ADE)
over those same records. Use the Workbench when you want a GUI for running,
supervising, reviewing, and landing tasks. Use `ctx work` when you want local
capture, redacted search, agent context packs, reviewer reports, schema
validation, import/export, and record inspection from the terminal.

ctx stores private state under `CTX_DATA_ROOT` or `~/.ctx` by default. Repo-local
`.ctx` files are explicit opt-in or legacy `ctx init` behavior, not the default
Work setup path.

## What ctx helps you do

- Keep coding-agent Work records local, durable, and inspectable.
- Link prompts, transcripts, tool activity, commands, diffs, PRs, artifacts, and checks.
- Review agent changes with the context that produced them.
- Reuse prior task context with future agents and human reviewers.
- Use installed harnesses on `PATH` where possible, with managed provider setup as an optional path.
- Open the same records in the desktop Workbench when a GUI run and review surface is useful.
- Run ADE-managed sessions in host or containerized modes; container disk/network controls apply to those ADE/containerized runs.

## Get started

Use one small, low-risk task to validate the whole loop end to end: install,
connect a provider, open a workspace, run a task, and review the resulting Work
record before you finalize changes.
