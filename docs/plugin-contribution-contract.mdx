# Plugin Contribution Contract

Plugins are installable bundles that contribute composable ctx surfaces. They
are not a replacement for the Work model, the ADE state model, or ACP.

## Contribution Types

The public contribution surface must be typed by the repo-local plugin SDK.

Supported or explicitly deferred contribution types:

| Contribution | First-pass expectation |
| --- | --- |
| Commands | Supported |
| ACP providers | Supported |
| Collectors/importers | Supported |
| Observers | Supported |
| Redaction/export processors | Deferred as SDK sidecar markers only |
| Workbench panels | Supported through `ui_surfaces` |
| Workbench templates | Supported as declarative host-owned manifest contributions |
| Card renderers | Supported as declarative host-owned manifest contributions |
| Detail sections | Supported as declarative host-owned manifest contributions |
| Review sections | Supported as declarative host-owned manifest contributions |
| Toolbar actions | Supported as declarative declared-plugin-command or approved-action references |
| Artifact renderers | Supported as declarative host-owned manifest contributions |
| Arbitrary webview/React UI | Deferred or gated behind explicit capability |

Silent partial support is not allowed. If a contribution appears in the public
manifest schema, the SDK must type it or the docs/tests must clearly mark it as
deferred.

This first declarative slice adds manifest buckets for host-owned Workbench
contributions that do not load or execute plugin UI code:

- `templates`;
- `toolbar_actions`;
- `artifact_renderers`;
- `card_renderers`;
- `detail_sections`;
- `review_sections`.

These declarations identify host-known templates, renderer IDs, sections, data
sources, and action targets. Toolbar `command` targets must reference commands
declared by the same plugin. They must not include `entrypoint`, JavaScript
module paths, webview URLs, React component names, or arbitrary renderer code.

## Host-Owned Workbench Vocabulary

The first public vocabulary is deliberately small and host-owned. Plugins may
declare that they want these surfaces; ctx decides how to render them.

Supported built-in Workbench templates:

- `classic`: default task/session workspace;
- `kanban`: task board plus selected task/session detail;
- `multipane`: resizable split panes over existing Workbench primitives;
- `review`: focused task review with Work metrics, PR links, change sets, and
  active task content;
- `review-summary`: plugin-compatible alias for the host review composition.

Supported host renderer IDs:

- `host.patch-artifact`: patch/diff artifact summary;
- `host.review-finding-card`: review finding card;
- `host.work-summary-detail`: Work summary/detail section;
- `host.review-gate-section`: local review/check evidence section.

Unsupported template or renderer IDs are still valid data, but the ADE must
render them as unsupported source-labeled contributions instead of executing
plugin code or silently dropping them. New host IDs should be added with schema,
SDK, Rust model, frontend projection, and visual coverage.

`ui_surfaces` remains the bucket for plugin-contributed surface placements such
as panels, sidebars, status bar entries, command palette entries, and settings.
Those surfaces may reference plugin entrypoints and are therefore distinct from
the host-owned declarative buckets above. Panels are intentionally not duplicated
as a new `panels` bucket in this slice.

Redaction and export processors remain deferred as SDK sidecar markers. They are
not manifest fields until processor provenance, permission prompts, redaction
defaults, and execution lifecycle are specified.

## Composability Rule

Workbench primitives are the composable internal units. Plugins contribute
configuration, bindings, renderers, and actions over those primitives.

A plugin should not mutate the store directly or replace the whole ADE shell in
the first pass.

Collector/importer plugins may produce Work event payloads, but they must submit
them through approved ctx import/capture actions. They must not receive direct
store write access.

## Data Bindings

Declarative UI contributions may bind to explicit, versioned data sources:

- current Work summary;
- current task/session/run;
- transcript slice;
- tool-call list;
- artifact list;
- check/gate state;
- change-set/diff summary;
- PR/contribution links;
- provider status;
- plugin diagnostics.

Bindings must define:

- loading state;
- empty state;
- error state;
- incremental update behavior;
- stale/reload behavior;
- unmount cleanup.

## Actions

Plugin UI can request approved actions by stable action ID and typed input.

Initial action candidates:

- focus Work/task/session;
- start task;
- run ctx command;
- run plugin command;
- export/redact Work;
- attach artifact;
- add manual note/attestation;
- update local gate state;
- open provider settings;
- request provider/session restart.

Action handlers own validation, permissions, provenance, redaction defaults, and
store writes.

## Capability And Trust Model

Local plugins are powerful local code. The first-pass trust model is:

- plugins only load from explicit plugin roots;
- dev-mode execution is source-labeled;
- command entrypoints, cwd, args, and env are validated;
- host process execution requires a visible plugin identity;
- arbitrary UI/webview execution is deferred or capability-gated;
- plugin stdout/stderr may be sensitive and must be bounded/redacted where
  displayed or exported;
- duplicate plugin IDs, provider IDs, and command IDs follow the collision rules
  below.

ID collision rules by class:

- plugin IDs are hard conflicts;
- provider IDs are hard conflicts unless a future explicit ownership/override
  mechanism exists;
- command IDs are source-qualified by owner and must not silently shadow provider
  commands;
- renderer, panel, and template IDs are plugin-qualified;
- local persisted references to removed plugin-qualified IDs must fall back with
  diagnostics rather than crashing.

## Hot Reload

Plugin reload must handle:

- add/change/remove of manifests;
- invalid manifest recovery;
- last-good revision behavior where safe;
- removed panels/templates and persisted fallback;
- in-flight command behavior;
- collector restart from cursor/checkpoint;
- provider removal without deleting historical Work;
- registry generation changes;
- diagnostics visible in CLI and ADE.

UI reload must not kill daemon-owned active sessions.

## Local CLI Dev Loop

The public local plugin workflow uses the single `ctx` binary:

- `ctx plugin validate <manifest-or-directory>`;
- `ctx plugin list [--root <path> ...] [--json]`;
- `ctx plugin reload [--root <path> ...] [--json]`.

`validate` reads a manifest file directly, or the first supported manifest name
inside a plugin directory (`ctx-plugin.json`, then `plugin.json`), parses it
through the Rust manifest model, and runs the same manifest validation used by
the daemon loader.

`list` and `reload` are local scanner commands. They do not contact or mutate a
running daemon. Their output is labeled `local_scan` so users can distinguish
offline diagnostics from daemon/UI hot reload. The root order is:

- explicit `--root` values;
- `CTX_PLUGIN_ROOTS`, with the same empty-entry normalization as the daemon;
- `<prepared CTX_DATA_ROOT or ~/.ctx>/plugins`.

The daemon-owned hot-reload path remains responsible for active-session safety,
local provider-adapter registry updates, last-good preservation, and ADE
updates. This is local daemon state only; it does not include hosted/team sync.
Future CLI commands that mutate a running daemon should use a different,
explicit verb or option so local scanner behavior stays unsurprising.

## SDK Shape

The repo-local SDK should start narrow and typed:

```ts
import { defineCtxPlugin } from "@ctx/plugin-sdk";

export default defineCtxPlugin({
  id: "example.review",
  name: "Example Review",
  version: "0.1.0",
  contributes: {
    commands: [],
    ui_surfaces: [],
    templates: [],
    toolbar_actions: [],
    artifact_renderers: [],
    card_renderers: [],
    detail_sections: [],
    review_sections: [],
  },
});
```

Publishing to npm is not required for the local program. The package should be
publishable later, but first it must work as a repo-local package with examples
and tests.

The public manifest/schema/SDK/Rust model/types contract is implemented for the
host-owned declarative buckets above. Daemon registry projection and frontend
projection now carry those buckets as source-labeled, inert Workbench
contribution data.

Still deferred:

- arbitrary plugin React/webview/module execution;
- daemon-connected plugin apply/reload commands that mutate a running daemon;
- plugin log streaming and dev-process lifecycle management;
- redaction/export processors as executable manifest fields.

## Worker Rule

Only the manager branch may change plugin capability semantics. Workers may add
contributions only after this contract and the manifest/SDK contract land in the
manager branch.
