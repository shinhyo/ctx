# Work Records

Work is the local record ctx keeps around coding-agent activity. A useful record
can include the objective, transcript, commands, tool activity, change sets,
artifacts, checks, and pull-request links that explain how a change happened.

The desktop Workbench, `ctx setup`, and the `ctx work` CLI use the same local
workspace registry and record model. The Workbench is the richer run and review
surface. The CLI is the terminal surface for setup, schemas, validation,
import/export, capture, PR linking, and local record inspection.

## Local Storage

By default, ctx stores private state under the prepared data root:

```text
${CTX_DATA_ROOT:-~/.ctx}
```

That data root contains the workspace registry, local stores, objects, plugin
state, and logs. Repo-local `.ctx` files are explicit opt-in or legacy `ctx init`
behavior; they are not the default Work setup location.

Use shareable exports when you want to move records. Safe exports should redact
host-local roots, raw transcript-like payloads, and obvious secrets unless you
intentionally choose a full local export for trusted import on the same machine
or another trusted machine.

## Current CLI Surface

Use `ctx setup` to register workspaces in the user-local data root:

```bash
ctx setup workspace .
ctx setup scratch
ctx setup status
ctx setup uninstall
```

By default setup installs ctx-owned `git` and `gh` shims under
`${CTX_DATA_ROOT:-~/.ctx}/bin`. Put that directory on `PATH` when you want
best-effort command capture. The shims forward to the real command first,
preserve its exit code, then record low-trust command context if ctx capture
succeeds. `ctx setup uninstall` removes only ctx-owned shims.

The current `ctx work` CLI supports local store and file operations plus narrow
capture/enrichment commands. Some older commands still read or write the v1
compatibility graph (`ChangeSet` and `Contribution`) while the observability
commands below operate on first-class `wrk_` Work Records.

```bash
ctx work schema
ctx work list
ctx work show <id>
ctx work capture command --tool git --exit-code 0 -- git status
ctx work link-pr https://github.com/owner/repo/pull/123
ctx work link-commit <sha>
ctx work note "Validated the linked PR locally"
ctx work recent
ctx work search "validation"
ctx work context <work-id> --json
ctx work report <work-id> --markdown
ctx work timeline <work-id>
ctx work evidence <work-id> run --kind test -- cargo test -p your-crate
ctx work evidence <work-id> freshness
ctx work summarize <work-id> --kind report
ctx work index rebuild --json
ctx work export --output work.json
ctx work import work.json --dry-run
ctx work validate work.json
ctx work inspect work.json
ctx work redaction-preview work.json
```

`ctx work list`, `ctx work show`, `ctx work export`, and `ctx work import`
currently operate on the compatibility graph of change sets and contributions.
They are kept so older local records remain inspectable and movable. Use
`ctx work search`, `ctx work context`, `ctx work report`, `ctx work timeline`,
`ctx work evidence`, `ctx work summarize`, `ctx work link-pr`, and
`ctx work link-commit` for first-class Work Record observability.

If `--workspace-id` is omitted, ctx first tries to select the registered
workspace that contains the current working directory. If that is ambiguous, use
`--workspace-id`. Use `--data-dir` to point at a specific ctx data root; if
omitted, ctx uses `CTX_DATA_ROOT`, then `~/.ctx`.

Best-effort shim capture can observe command argv, cwd, Git facts, exit status,
and deterministic PR references such as explicit GitHub PR URLs or
`gh pr view 123 --repo owner/repo`. It cannot infer every intent or every PR
created by a command. Use `ctx work link-pr` when the PR URL is known.

## Observe, Search, Context, Report

The P0 Work observability surface is local and CLI-first:

- `ctx work search` queries the redacted local search index.
- `ctx work context <work-id> --json` returns a bounded agent context pack with
  citations, evidence freshness, and no raw transcript payloads.
- `ctx work report <work-id> --markdown` renders a reviewer-facing summary of
  trust, evidence, linked changes, and the redacted timeline.
- `ctx work evidence <work-id> run -- ...` records an observed local command,
  its exit status, Git fingerprint, and redacted bounded output.
- `ctx work evidence <work-id> freshness` compares stored evidence fingerprints
  to the current checkout and marks evidence fresh, stale, partial, or unknown.
- `ctx work summarize` creates deterministic local summaries with citation
  claims; summaries become stale when the underlying Work material changes.
- `ctx work index rebuild --json` rebuilds redacted search docs from local Work
  records.

Evidence means "observed local command/artifact," not proof. A fresh passing
evidence record can support a `verified` local trust verdict, but shimmed command
capture remains low-trust context unless it is backed by fresh evidence and a
durable PR or commit link.

The daemon exposes the same local Work report/context/timeline/evidence data to
the desktop UI through redacted workspace-scoped routes. Those routes are
full-daemon-token desktop routes for now; scoped MCP/agent access to Work records
is intentionally not part of this P0 public slice.

## Agent Usage

Agents should treat `ctx work ... --json` as the stable local contract:

1. Search before starting or resuming a task:

   ```bash
   ctx work search "sandbox launch failure" --json
   ctx work search --pr https://github.com/owner/repo/pull/123 --json
   ctx work search --commit abc123 --json
   ```

2. Select a `work_id` from the result, then request bounded context:

   ```bash
   ctx work context <work-id> --budget 12000 --json
   ```

3. Link durable source-control anchors when they become known:

   ```bash
   ctx work link-pr https://github.com/owner/repo/pull/123
   ctx work link-commit <sha>
   ```

4. Record and refresh evidence instead of claiming validation from memory:

   ```bash
   ctx work evidence <work-id> run --kind test -- cargo test -p your-crate
   ctx work evidence <work-id> freshness --json
   ```

5. Treat stale or missing evidence as a rerun-or-warn condition. Do not present
   user-space shim capture, manual notes, or imported summaries as proof.

`verified` requires fresh evidence with verified provenance. Fresh local
worktree evidence is useful, but it can still be only `partial` if provenance is
not independently verified.

## Desktop/API Route Surface

The desktop app uses daemon-token, local, workspace-scoped routes over the same
records. They are not a hosted API and are not scoped MCP tools.

```text
GET  /api/workspaces/:id/work
GET  /api/workspaces/:id/work/:work_id
GET  /api/workspaces/:id/work/:work_id/context
GET  /api/workspaces/:id/work/:work_id/report
GET  /api/workspaces/:id/work/:work_id/timeline
GET  /api/workspaces/:id/work/:work_id/evidence
POST /api/workspaces/:id/work/:work_id/evidence
POST /api/workspaces/:id/work/:work_id/summaries
```

Default route responses are redacted and bounded. Raw transcript payloads,
arbitrary local files, host roots, and provider-backed summaries are not exposed
by default. The mutation routes record local observed evidence or local/manual
summaries; provider-backed LLM summary creation is rejected in this public local
slice.

## Record Shape

The v1 compatibility schema still uses `agent-work` names for files, API types,
and schema IDs. Public product copy should say Work. Treat `agent-work` as the
compatibility namespace until the migration can be done as one planned change.

The most important record types are:

- `AgentWork`: a workspace aggregate that contains change sets and contributions.
- `ChangeSet`: a coherent Git diff, patch, commit, or PR-bound unit of review.
- `Contribution`: a relationship or event that connects sessions, runs,
  artifacts, checks, change sets, and pull requests.
- `WorkRecord`: the durable task/change/review identity that links sessions,
  PRs, commits, evidence, summaries, and events.
- `WorkEvidence`: an observed validation artifact or command result with
  freshness relative to a recorded Git fingerprint.
- `WorkSummary` and `WorkSummaryClaim`: deterministic local summaries and
  cited claims used by reports and agent context packs.
- `WorkSearchDoc`: a redacted local search document; raw transcript payloads are
  not indexed by default.

Command captures and imported observations are useful context, but they are not
tamper-proof audit by themselves. Records should preserve source, fidelity, and
trust distinctions so local evidence is not overstated.
